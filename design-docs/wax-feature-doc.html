Product Feature Document — v1.0

Wax

A real-time, song-by-song album rating tool for intentional listening.

Author Victor Quan
Status Pre-development
Target Personal tool → public app
Stack Spring Boot · React · PostgreSQL
01

Overview

Wax is a lightweight web app that gamifies the album listening experience by prompting users to rate each track as they finish it — capturing an honest, in-the-moment reaction rather than a retrospective score shaped by memory or bias.

The core insight is simple: most music rating platforms (RateYourMusic, Album of the Year, Last.fm) are manual, retroactive, and disconnected from the actual act of listening. Wax is none of those things. It integrates with Spotify to detect what you're listening to in real time, prompts you after each song, and delivers a calculated final score when the album ends — a receipt for the listening session.

Built for personal use first. Designed to grow.

02

The problem

Streaming has eroded the album format. Shuffle culture and algorithmic playlists fragment records into individual songs, stripping them of the narrative arc, pacing, and sequencing that artists deliberately construct.

Even for listeners who still consume albums in full, the act of rating them is broken. You either assign a score days later — after the dopamine of new music has faded and a single standout track has skewed your whole perception — or you use a platform that demands you rate 15 songs from memory in one sitting.

The gap: No tool currently ties real-time Spotify playback to a track-by-track prompted rating system. That combination is what Wax is built around.

03

Phased roadmap

Phase 1 — Current

Personal tool

A standalone app (separate repo, Railway service) for solo use. Spotify OAuth, real-time now-playing detection via polling, track-by-track rating prompts, album scorecard at the end. PostgreSQL-backed history. Linked from portfolio as a project.

Phase 2 — Near-term

Polish & shareability

Public scorecard pages per session (shareable link). Personal album history with aggregate scores over time. A visual "album report card" designed for sharing. Optional notes per track. Genre and artist stats.

Phase 3 — Future

Multi-user & social

Public user auth (JWT, already built in Fantasy Seers). Community scores vs personal scores. Friends' ratings. A feed of recent sessions. Album discovery driven by friend activity.

04

Phase 1 — Feature list

Spotify integration

  • OAuth login — user authenticates via Spotify, token stored and refreshed server-side
  • Now-playing detection — polls Spotify's /me/player/currently-playing endpoint every 5–10 seconds to detect the active track and album
  • Track change detection — compares current track ID to previously polled state; triggers rating prompt on change
  • Album context — reads album name, artwork, track list, and total track count from the Spotify response to frame the session

Session management

  • Session start — intentional, not automatic — the user manually hits a "Start Session" button in Wax when they sit down to listen to an album. Wax reads whatever is currently playing from Spotify at that moment and anchors the session to that album. This eliminates false sessions from shuffle, radio, or mid-album resumption.
  • Session state — tracks which songs have been rated vs pending within the active session; displayed as a progress indicator in the UI
  • Session end — triggered by completion — a session is marked COMPLETED and the final scorecard is surfaced automatically when all tracks on the album have been rated (or skipped). There is no manual end trigger.
  • Incomplete sessions — if the user closes the app mid-album, the session is preserved with status IN_PROGRESS and can be resumed from the Wax home screen

Rating UX

  • Post-track prompt — a minimal, non-disruptive overlay or panel appears after a track change, prompting the user to rate the previous song
  • 1–10 scale — simple numeric rating; no half-points in v1 to keep interaction fast
  • Skip option — user can skip rating a track (it is excluded from the average)
  • Track metadata — prompt displays album art, track name, and track number for context
  • Optional note — a single free-text line the user can optionally add per track (deferred to Phase 2 if it adds friction)

Scorecard

  • Final album score — average of all rated tracks, displayed prominently at session end
  • Track breakdown — ordered list of every track with its individual score
  • Highlight / lowlight — highest and lowest rated track called out visually
  • Session history — past scorecards accessible from a personal history page

06

Tech stack

Wax re-uses the existing portfolio infrastructure. No new tools needed for v1.

Frontend — React + Vite
Backend — Spring Boot (Java)
Database — PostgreSQL on Railway
Auth — Spotify OAuth 2.0
Spotify API — polling now-playing endpoint
Deployment — Railway (new service)
Migrations — Flyway
Caching — Caffeine (token + album metadata)

Key technical constraint: Spotify does not offer webhooks for playback events. Track change detection relies on polling the /me/player/currently-playing endpoint every ~5 seconds. This is rate-limit safe for personal use but requires consideration before scaling to many users.

07

Design principles

01
Minimum friction above all

Rating a song should take under 5 seconds. Any interaction that pulls the user out of listening for longer than that is a failure.

02
The app should feel barely there

Wax is a companion to the listening session, not the main event. It lives in the background and surfaces only when needed.

03
Honest data over pretty data

The value of real-time ratings is their rawness. The UX must not encourage second-guessing or retroactive editing of scores.

04
Build for one, design for many

Every data model decision should account for multi-user expansion. No technical debt that requires a rewrite to go public.

08

Open questions for v1

Decided

  • Session start — ✓ Intentional. User manually presses "Start Session" in Wax; no auto-detection of listening activity.
  • Session end — ✓ Triggered automatically when all tracks on the album have been rated or skipped. No manual end action required.

Still open

  • Rating prompt timing — prompt fires when Wax detects a track change via polling. Should there be a short delay (e.g. 2–3 seconds) to let the new song settle before the prompt appears, or is immediate fine?
  • Retroactive edits — can the user change a rating after submitting it within the same session? Leaning yes for v1, but needs an explicit decision as it affects the service layer.
  • Mobile UX — primarily a desktop tool (browser tab alongside Spotify desktop) or responsive enough for mobile too? Affects layout decisions from the start.
